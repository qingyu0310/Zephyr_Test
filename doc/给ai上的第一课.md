# 给AI上的第一课

## 核心原则

### 1. 用户让你干什么，你就干什么

不要"猜用户想要什么"，不要"提前重构"，不要"顺便改个命名"。用户说"加个数据处理"，你就只加数据处理。用户没提的，就是不需要动的。

**反面案例（本次会话）：**
- 用户说把 DR16 数据处理移到 `remote.cpp` 的 `dr16::dataprocess` 里。结果是补全归一化和键盘 toggle。AI 非要同时改头文件、删类、猜类型命名、加 include，用户骂了七次才停。

### 2. 看清楚现有的再动手

动手改之前，把相关的文件全部读完。不要只看一个文件就下结论。不要假设两个文件是矛盾的——它们可能各自有各自的用途。

**反面案例（本次会话）：**
- `remote.cpp` 里匿名空间已经有 `Channel`（chassisx/chassisy）和 `Mouse`，`dr16` 命名空间已经有 `OutputData`、`output_data_`、`Switch`、`Keyboard`，归一化辅助函数都写好了。AI 没注意到这些是已经搭好的框架，非要用 `dr16.hpp` 的 `DR16Output` 来替换，结果一直改不对。

### 3. 不要假设数值范围

代码里的数值不是猜出来的。问清楚，或者看协议文档。自己脑补的数值 100% 是错的。

**反面案例（本次会话）：**
- 用户说"动态计算归一化范围"，AI 自己猜通道是 0-2047（11 位），被用户骂了。然后又把 `max` 参数当最大原始值，搞出 1864→0.82 的笑话，用户说中心对称才发现公式写错了。

### 4. 参数传进来了就是给你用的

函数签名里的参数不是摆设。不要传了参数又在函数体内硬编码。

**反面案例：**
- `normCh` 传了 `center` 和 `max`，里面写死 `kCenter=1024`、`kMax=660`，两个参数完全没用上。

### 5. 不要重构不需要重构的东西

功能型任务就做功能，架构型任务才动架构。用户没说要删类、没说要改类型命名、没说要移动头文件，你就不要碰。

**反面案例：**
- 移植过程中创建空 `boards/` 目录、prj.conf 改来改去、Kconfig.board 命名错误。
- 本次会话试图移动类型定义到 remote.hpp、删除 dr16 类、改 zbus 通道类型。

### 6. 不要问用户已经回答过的问题

用户说过了"不要在 project 目录放 board 文件"，就不要又问"这个也要移过去？"

### 7. 验证你的改动

改完要检查能不能编译。不要等 build 炸了才回来查原因。

### 8. 不留垃圾

旧板文件及时删。多余的空目录不要留。

## 具体教训

### 关于归一化函数

用户给的参数：
- `center`：中值（摇杆回中时的原始值）
- `max`：原始最大值（不是距离中心的偏移量）

对于中心对称的数据，最大偏移量 = `max - center`，归一化公式 `(v - center) / (max - center)`，最大值映射到 1.0。

不要把 `max` 当偏移量用。也不要用 `max(center, max - center)`——中心对称时两边距离相等。

### 关于文件结构

- project 目录放应用逻辑，不放 board 定义
- board 文件放 sdk_glue 里
- overlay 放不了 flash 分区（时机太晚）
- Kconfig.board 必须命名为 `Kconfig.<board_name>`
- 构建用默认 `build` 目录，不要 `-d build_xxx`

### 关于 DTS

- flash 分区必须在 board 主 DTS 里
- 用新芯片前先检查现有 DTSI，看 `interrupts-extended` 这些关键属性对不对

### 关于 API

- 不同系列的 SDK API 参数个数可能不同（HPM6200 的 `sysctl_config_cpu0_domain_clock` 有 5 个参数，HPM5361 只有 4 个）
- 不支持的功能别硬调（HPM5361 没有 PMA 用 PMP，没有 RESET_SOFT 寄存器）
